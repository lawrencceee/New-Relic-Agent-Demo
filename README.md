# Full-Stack Observability Lab: Synthetic Traffic & Performance Monitoring

## 📋 Project Overview
This project is a technical demonstration of a modern observability stack designed to simulate real-world e-commerce traffic and monitor system health. Using **New Relic**, this lab provides deep visibility into application performance (APM), infrastructure health, and user behavior.

The core objective of this project is to demonstrate how to establish a performance baseline and identify system bottlenecks under varying load conditions using a "Synthetic-First" approach.

---

## 🏗️ Architecture
The system follows a containerized micro-service architecture consisting of four distinct layers:

1.  **Synthetic Traffic Layer (Locust):** Acting as the "Workload Simulator," this layer generates concurrent user behaviors and edge-case scenarios.
2.  **Application Layer (FastAPI):** A high-performance Python-based REST API instrumented with the New Relic Python Agent.
3.  **Persistence Layer (SQLite):** Local storage used to simulate database interactions and track SQL query performance/latency.
4.  **Observability Layer (New Relic):** The central nervous system for telemetry ingestion, correlating APM traces with Host-level metrics.

## 🚀 Key Technical Insights

### 1. The "Signal-to-Noise" Strategy
Rather than generating random requests, I configured the **Locust** layer to prioritize "High-Value Transactions" (e.g., `/checkout` vs `/search`). This ensures that New Relic dashboards reflect business-critical data rather than raw, uninterpreted telemetry.

### 2. Infrastructure Gaps (The "Mac Virt" Challenge)
During development on macOS, I identified a specific telemetry gap where the standard Infrastructure Agent could not natively "see" container-level resource usage due to the macOS virtualization layer. I resolved this by utilizing New Relic’s Docker-specific instrumentation to bridge the host-container gap.

### 3. Distributed Tracing
The application is configured to provide full-stack visibility. Every request generated by Locust is traceable from the entry point down to the specific SQL `SELECT/INSERT` statement, allowing for rapid Mean Time to Detection (MTTD).

---

## 📈 Business Outcomes
By implementing this observability stack, we transition from reactive monitoring to proactive optimization:
* **MTTR Reduction:** Immediate correlation between code-level errors and infrastructure resource spikes.
* **Persona-Based Dashboards:** Data is structured to serve both DevOps (latency/errors) and Business stakeholders (transaction success rates).
* **Predictive Scaling:** Using synthetic load to identify the exact "breaking point" of the infrastructure before real users are impacted.

## ⚙️ Setup & Installation

### 1. Prerequisites
* Docker and Docker Compose installed.
* A New Relic account and an **Ingest License Key**.

### 2. Configuration
Create a `.env` file in the root directory:
```bash
NEW_RELIC_LICENSE_KEY=your_license_key_here
NEW_RELIC_APP_NAME="FastAPI-Observability-Lab"
```
